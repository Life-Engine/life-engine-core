# Document Storage Trait

## Overview

The document storage trait defines the contract that all document storage adapters must implement. It covers structured data operations — CRUD, querying, batch operations, transactions, change watching, migration, and health checks.

Adapters are native Rust trait implementations. V1 ships one built-in adapter (SQLite/SQLCipher). The trait is designed to be future-proof — additional adapters can be implemented without changing the trait signature.

## Trait Definition

```rust
#[async_trait]
pub trait DocumentStorageAdapter: Send + Sync {
    // Single-document operations
    async fn get(&self, collection: &str, id: &str) -> Result<Document, StorageError>;
    async fn create(&self, collection: &str, doc: Document) -> Result<String, StorageError>;
    async fn update(&self, collection: &str, id: &str, doc: Document) -> Result<(), StorageError>;
    async fn partial_update(&self, collection: &str, id: &str, patch: Value) -> Result<(), StorageError>;
    async fn delete(&self, collection: &str, id: &str) -> Result<(), StorageError>;

    // Query operations
    async fn list(&self, collection: &str, query: QueryDescriptor) -> Result<DocumentList, StorageError>;
    async fn count(&self, collection: &str, query: QueryDescriptor) -> Result<u64, StorageError>;

    // Batch operations
    async fn batch_create(&self, collection: &str, docs: Vec<Document>) -> Result<Vec<String>, StorageError>;
    async fn batch_update(&self, collection: &str, updates: Vec<(String, Document)>) -> Result<(), StorageError>;
    async fn batch_delete(&self, collection: &str, ids: Vec<String>) -> Result<(), StorageError>;

    // Transaction
    async fn transaction<F, R>(&self, f: F) -> Result<R, StorageError>
    where
        F: FnOnce(&dyn TransactionHandle) -> Result<R, StorageError> + Send;

    // Change watching
    fn watch(&self, collection: &str) -> Pin<Box<dyn Stream<Item = ChangeEvent> + Send>>;

    // Lifecycle
    async fn migrate(&self, descriptor: CollectionDescriptor) -> Result<(), StorageError>;
    async fn health(&self) -> HealthReport;
    fn capabilities(&self) -> AdapterCapabilities;
}
```

## Single-Document Operations

- **get** — Retrieve a single document by collection and ID. Returns `StorageError::NotFound` if the document does not exist.
- **create** — Insert a new document. Returns the assigned ID. Returns `StorageError::AlreadyExists` if a document with the same ID already exists.
- **update** — Full replacement of a document by collection and ID. The entire document body is replaced. Returns `StorageError::NotFound` if the document does not exist.
- **partial_update** — Patch specific fields. Accepts a JSON value containing only the fields to update. Adapters that support native partial update (reported via capabilities) apply it directly. Adapters that do not support it receive a full document from `StorageContext`, which performs a read-modify-write internally.
- **delete** — Remove a document by collection and ID. Returns `StorageError::NotFound` if the document does not exist.

## QueryDescriptor

The `QueryDescriptor` is a backend-agnostic query representation produced by `StorageContext`'s fluent API:

```rust
pub struct QueryDescriptor {
    pub filters: Option<FilterNode>,
    pub sort: Vec<SortField>,
    pub pagination: Pagination,
    pub fields: Option<Vec<String>>,
    pub text_search: Option<String>,
}
```

### Filters

Filters are composable as a tree of nodes:

```rust
pub enum FilterNode {
    Condition {
        field: String,
        operator: FilterOperator,
        value: Value,
    },
    And(Vec<FilterNode>),
    Or(Vec<FilterNode>),
}

pub enum FilterOperator {
    Eq,
    Neq,
    Gt,
    Gte,
    Lt,
    Lte,
    Contains,
    StartsWith,
    Exists,
    In,
}
```

- `field` uses dot-notation for nested access (e.g., `address.city`, `ext.connector-email.thread_id`)
- `In` matches any value in a provided list
- `Exists` checks whether a field is present (value is ignored)
- Filters can be nested arbitrarily: `And([condition, Or([condition, condition])])`

### Sort

```rust
pub struct SortField {
    pub field: String,
    pub direction: SortDirection,
}

pub enum SortDirection { Asc, Desc }
```

Multiple sort fields are applied in order.

### Pagination

Two modes:

```rust
pub enum Pagination {
    Offset { offset: u64, limit: u64 },
    Cursor { cursor: String, limit: u64 },
}
```

- **Offset-based** — Simple, works with every adapter.
- **Cursor-based** — The cursor is an opaque string generated by the adapter. Adapters that do not support cursors (reported via capabilities) encode the offset into the cursor as a fallback.

### Field Projection

An optional list of fields to return. If omitted, the full document is returned. Adapters that cannot do projection at the storage level ignore this — `StorageContext` strips fields before returning to the caller.

### Text Search

An optional full-text search query. Adapters that support text search (reported via capabilities) use their native implementation (e.g., SQLite FTS5). Adapters that do not fall back to `Contains` matching on string fields.

## Batch Operations

- **batch_create** — Insert multiple documents. Returns a list of assigned IDs in the same order as the input. `StorageContext` wraps batch operations in a transaction internally.
- **batch_update** — Replace multiple documents. Each entry is a `(id, document)` pair. All-or-nothing within a transaction.
- **batch_delete** — Remove multiple documents by ID. All-or-nothing within a transaction.

Batch operations exist for sync connectors that pull many records at once. They are the plugin-facing abstraction for atomic multi-document writes — plugins do not get direct transaction access.

## Transaction

The transaction method accepts a closure that receives a `TransactionHandle`. The handle exposes the same single-document operations as the trait:

```rust
#[async_trait]
pub trait TransactionHandle: Send + Sync {
    async fn get(&self, collection: &str, id: &str) -> Result<Document, StorageError>;
    async fn create(&self, collection: &str, doc: Document) -> Result<String, StorageError>;
    async fn update(&self, collection: &str, id: &str, doc: Document) -> Result<(), StorageError>;
    async fn partial_update(&self, collection: &str, id: &str, patch: Value) -> Result<(), StorageError>;
    async fn delete(&self, collection: &str, id: &str) -> Result<(), StorageError>;
}
```

- If the closure returns `Ok`, the adapter commits
- If the closure returns `Err` or panics, the adapter rolls back
- Transactions can span multiple collections within the same adapter
- Adapters that do not support transactions execute operations sequentially and attempt manual rollback on failure

Plugins do not get direct transaction access. `StorageContext` uses transactions internally for batch operations. This prevents WASM plugins from holding locks indefinitely.

## Change Watching

The `watch` method returns a stream of change events for a collection:

```rust
pub struct ChangeEvent {
    pub collection: String,
    pub id: String,
    pub operation: ChangeOperation,
    pub changed_fields: Option<Vec<String>>,
    pub timestamp: DateTime<Utc>,
}

pub enum ChangeOperation { Created, Updated, Deleted }
```

Adapters that support native change detection (e.g., Postgres LISTEN/NOTIFY) push events through this stream. Adapters that do not support native watch return an empty stream — `StorageContext` handles event emission on the write path instead.

## Migration

The `migrate` method receives a `CollectionDescriptor` and ensures the adapter's storage is ready for that collection:

```rust
pub struct CollectionDescriptor {
    pub name: String,
    pub owner: CollectionOwner,
    pub fields: Vec<FieldDescriptor>,
    pub indexes: Vec<String>,
    pub schema_version: String,
}

pub enum CollectionOwner {
    Shared,
    Plugin(String),
}

pub struct FieldDescriptor {
    pub path: String,
    pub field_type: FieldType,
}

pub enum FieldType { String, Number, Boolean, Array, Object, DateTime }
```

Migration behaviour:

1. If the collection does not exist, create it
2. If the collection exists, compare the current descriptor with the stored one
3. Apply additive changes automatically (new fields, new indexes, new collections)
4. Return `StorageError::SchemaConflict` for breaking changes (removed fields, changed types)
5. Store the current descriptor for future comparison

Migration is **idempotent** — it runs every startup for every known collection. If nothing changed, it is a no-op.

The `fields` list is used for storage optimisation, not enforcement. An adapter can use field descriptors to create typed columns or generated columns for indexed JSON fields. A minimal adapter ignores them and stores everything as JSON.

## Health Check

The `health` method returns a `HealthReport`:

```rust
pub struct HealthReport {
    pub status: HealthStatus,
    pub backend_name: String,
    pub checks: Vec<HealthCheck>,
}

pub enum HealthStatus { Healthy, Degraded, Unhealthy }

pub struct HealthCheck {
    pub name: String,
    pub passed: bool,
    pub message: Option<String>,
}
```

Document adapter health checks:

- Connection alive (can execute a trivial query)
- Storage accessible (read/write permissions)
- Encryption active (if capability declared)
- Collections exist (all expected collections are present)

At startup, both adapters must return `Healthy` or Core refuses to start. `Degraded` is a runtime-only state for situations like low disk space or connection pool pressure.

## Adapter Capabilities

Adapters declare what they support via a capabilities struct:

```rust
pub struct AdapterCapabilities {
    pub encryption: bool,
    pub indexing: bool,
    pub transactions: bool,
    pub native_watch: bool,
    pub cursor_pagination: bool,
    pub partial_update: bool,
    pub text_search: bool,
    pub field_projection: bool,
}
```

For capabilities an adapter does not support, `StorageContext` or the [[storage-router|StorageRouter]] provides fallback behaviour:

- No `partial_update` → `StorageContext` does read-modify-write
- No `native_watch` → `StorageContext` emits events on the write path
- No `cursor_pagination` → Offset encoded as opaque cursor string
- No `text_search` → `Contains` matching on string fields
- No `field_projection` → `StorageContext` strips fields after retrieval
- No `transactions` → Sequential execution with manual rollback
- No `indexing` → Index hints ignored, adapter scans

Required capabilities are configurable in `storage.toml`. Core validates at startup that the active adapter meets the requirements. Default configuration requires `encryption` for the document adapter.

## Error Model

All trait methods return `StorageError` on failure:

```rust
pub enum StorageError {
    NotFound { collection: String, id: String },
    AlreadyExists { collection: String, id: String },
    ValidationFailed { errors: Vec<FieldError> },
    CapabilityDenied { capability: String, plugin_id: String },
    CollectionNotFound { collection: String },
    SchemaConflict { collection: String, message: String },
    Timeout { operation: String, duration_ms: u64 },
    BackendError { message: String, code: Option<String>, retryable: bool },
    TransactionFailed { reason: String },
}
```

Mapping to `WorkflowStatus`:

- `NotFound` → `WorkflowStatus::NotFound`
- `AlreadyExists`, `ValidationFailed`, `SchemaConflict` → `WorkflowStatus::Invalid`
- `CapabilityDenied` → `WorkflowStatus::Denied`
- `Timeout`, `BackendError`, `TransactionFailed` → `WorkflowStatus::Error`

The `retryable` flag on `BackendError` indicates whether retrying the operation could succeed (e.g., transient connection failure vs corrupt data). The workflow layer's `on_error: retry` mechanism can check this flag to avoid wasting retry attempts on non-recoverable errors.
