# Storage Context

## Overview

`StorageContext` is the API surface that plugins and workflows use to interact with storage. It sits between callers and the [[storage-router|StorageRouter]], enforcing permissions, validating schemas, scoping collections, emitting audit events, and bridging storage change events to the [[architecture/core/design/workflow-engine-layer/event-bus|event bus]].

Plugins never interact with adapters directly. All storage access flows through `StorageContext`.

## Two Access Paths

`StorageContext` serves two types of callers with different trust levels:

- **Plugin access (via host functions)** — A plugin calls a host function (e.g., `storage_doc_get`). The host function creates a `StorageContext` scoped to that plugin's identity. Permission checks and collection scoping are enforced.
- **Workflow engine access (internal)** — System workflows and the pipeline executor call `StorageContext` directly with a system-level identity. System identity bypasses plugin-scoped permission checks. Schema validation still applies.

## Permission Enforcement

`StorageContext` checks capabilities before any operation reaches the adapter:

- **storage:doc:read** — Required for get, list, count, query on document storage
- **storage:doc:write** — Required for create, update, partial_update, and batch create/update on document storage
- **storage:doc:delete** — Required for delete and batch delete on document storage
- **storage:blob:read** — Required for retrieve, exists, list, metadata on blob storage
- **storage:blob:write** — Required for store and copy on blob storage
- **storage:blob:delete** — Required for delete on blob storage

Capabilities are per-collection, declared in the plugin manifest. A plugin that declares `read-write` access to `contacts` gets `storage:doc:read` and `storage:doc:write` scoped to the `contacts` collection only.

If a plugin attempts an operation it lacks the capability for, `StorageContext` returns `StorageError::CapabilityDenied` immediately. The request never reaches the adapter.

## Collection Scoping

Plugins can only access collections they have declared in their manifest:

- **Shared collections** (CDM or otherwise) — Declared with an access level (`read`, `write`, `read-write`). Multiple plugins can share these.
- **Plugin-scoped collections** — Automatically namespaced to the plugin. A plugin's collection `shopping_lists` is stored as `{plugin_id}.shopping_lists` to prevent collisions.

`StorageContext` enforces this boundary. A plugin cannot query a collection it has not declared, regardless of capabilities.

System-level callers (system workflows) can access any collection.

## Schema Validation

`StorageContext` validates documents on write operations (create, update, partial_update, and their batch variants) before passing them to the adapter:

- **Collections with a declared schema** (CDM or custom) — Validated against the JSON Schema referenced in the plugin manifest
- **Collections without a declared schema** — No validation, store anything

Validation is opt-in per collection. If a plugin declares a schema in its manifest, data is validated against it. If not, data is stored as-is.

Validation is **permissive by default** — extra fields not in the schema are accepted and stored but not indexed or validated. Plugins can opt into strict mode per collection in their manifest, which rejects unknown fields.

Read operations are never validated. Data is trusted once stored.

See [[schema-and-validation]] for full details on schema format and validation rules.

## System-Managed Base Fields

`StorageContext` ensures every document in every collection has required base fields:

- `id` — Unique identifier. Generated by `StorageContext` on create if not provided.
- `created_at` — Timestamp. Set by `StorageContext` on create.
- `updated_at` — Timestamp. Set by `StorageContext` on every update.

These fields are guaranteed to exist on every document regardless of what the plugin provides or whether the collection has a declared schema.

## Extension Field Handling

Plugins can attach extra fields to shared collection documents. `StorageContext` namespaces these to prevent collisions:

- Extension fields are stored under `ext.{plugin_id}.{field_name}`
- A plugin declaring extensions in its manifest can read and write its own extension fields
- Other plugins see extension fields but cannot modify them
- Extension fields can have their own schema validation if declared in the manifest

## Query Building

`StorageContext` provides a fluent API that plugins use to construct queries. The API produces a backend-agnostic `QueryDescriptor` that the [[storage-router|StorageRouter]] passes to the adapter:

```rust
ctx.collection("contacts")
   .filter(|f| f.field("address.city").eq("London"))
   .sort("updated_at", Desc)
   .limit(20)
   .cursor(prev_cursor)
   .exec()
```

See [[document-storage-trait#QueryDescriptor]] for the full descriptor specification.

## Host Functions

The following host functions are exposed to WASM plugins:

Document operations:

- `storage_doc_get(collection, id) → Result<Document>`
- `storage_doc_list(collection, query_json) → Result<DocumentList>`
- `storage_doc_count(collection, query_json) → Result<u64>`
- `storage_doc_create(collection, doc_json) → Result<Id>`
- `storage_doc_update(collection, id, doc_json) → Result<()>`
- `storage_doc_partial_update(collection, id, patch_json) → Result<()>`
- `storage_doc_delete(collection, id) → Result<()>`
- `storage_doc_batch_create(collection, docs_json) → Result<Vec<Id>>`
- `storage_doc_batch_update(collection, updates_json) → Result<()>`
- `storage_doc_batch_delete(collection, ids_json) → Result<()>`

Blob operations:

- `storage_blob_store(key, bytes) → Result<()>`
- `storage_blob_retrieve(key) → Result<Bytes>`
- `storage_blob_delete(key) → Result<()>`
- `storage_blob_exists(key) → Result<bool>`
- `storage_blob_list(prefix) → Result<Vec<BlobMeta>>`
- `storage_blob_metadata(key) → Result<BlobMeta>`

Not exposed to plugins: `transaction`, `watch`, `migrate`, `health`, `copy`, `query` (accessed through `list` with a query descriptor).

## Audit Event Emission

`StorageContext` emits audit events for **write operations only** via the [[architecture/core/design/workflow-engine-layer/event-bus|event bus]]:

- `system.storage.created` — `{ collection, id }`
- `system.storage.updated` — `{ collection, id, changed_fields }`
- `system.storage.deleted` — `{ collection, id }`
- `system.blob.stored` — `{ key }`
- `system.blob.deleted` — `{ key }`

Each audit event includes the plugin ID (or `"system"` for system workflows), the collection or blob key, and the operation type. Full document payloads are not included in audit events.

Read operations are not audited. This keeps audit logging focused and avoids performance overhead on the hot path.

## Watch-to-Event-Bus Bridge

`StorageContext` bridges storage change events from the active document adapter to the event bus:

1. At startup, `StorageContext` subscribes to the document adapter's `watch` stream
2. Backend change events are translated into the `system.storage.*` events listed above
3. Workflows can trigger on these events like any other

If the adapter supports native watch (reported via capabilities), `StorageContext` uses the adapter's stream. If not, `StorageContext` emits events directly on the write path — it sees every write pass through, so no separate mechanism is needed.

No duplicate events are emitted. If the adapter provides native watch, `StorageContext` does not also emit on the write path.

## Credential Encryption

The `credentials` collection receives additional treatment. `StorageContext` encrypts sensitive fields (OAuth tokens, API keys) individually with a derived key before passing them to the document adapter. This is independent of adapter-level encryption and ensures credentials are protected even if the adapter's encryption is compromised.

See [[encryption-and-audit]] for key derivation and encryption details.

## Error Handling

`StorageContext` catches all `StorageError` variants from the adapter and either:

- Returns them to the calling workflow step (which handles via `on_error`)
- Translates them into host function error responses for plugins

See [[document-storage-trait#Error Model]] for the full error type definition and mapping to `WorkflowStatus`.
