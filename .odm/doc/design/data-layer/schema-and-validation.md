# Schema and Validation

## Overview

Schemas define the expected shape of documents in a collection. Validation is performed by [[storage-context|StorageContext]] on write operations before data reaches the adapter. Adapters store whatever they receive — they do not validate.

This document covers schema format, where schemas are declared, validation rules, system-managed base fields, extension field namespacing, and index hints.

## Schema Format

Schemas use **JSON Schema** (draft 2020-12). JSON Schema is a well-known standard with language-agnostic tooling and mature Rust validation libraries.

Plugins ship their schema files alongside `manifest.toml` in their plugin directory:

```
plugins/
  connector-email/
    plugin.wasm
    manifest.toml
    schemas/
      email-thread.json
```

## Where Schemas Are Declared

### CDM Recommended Schemas

The SDK ships recommended schemas for common collections (events, tasks, contacts, notes, emails, credentials). These are JSON Schema files that plugins can adopt by referencing them in their manifest.

CDM schemas are versioned with the SDK — see [[schema-versioning-rules]] for the full versioning policy.

### Plugin Collections

All collections are declared in `manifest.toml`. Plugins choose whether to adopt a CDM schema, provide a custom schema, or use no schema at all:

```toml
# A collection using a CDM recommended schema
[collections.contacts]
schema = "cdm:contacts"
access = "read-write"
extensions = ["ext.connector-google.google_id", "ext.connector-google.last_sync"]
extension_schema = "schemas/contact-extensions.json"
extension_indexes = ["ext.connector-google.google_id"]

# A collection with a custom schema
[collections.shopping_lists]
schema = "schemas/shopping-list.json"
indexes = ["due_date", "category"]
strict = false

# A schemaless collection
[collections.cache]
```

- `schema` — Path to a JSON Schema file (relative to plugin directory), or a `cdm:` prefix to adopt a CDM recommended schema. Optional — if omitted, the collection is schemaless.
- `access` — One of `read`, `write`, or `read-write`. Maps to `storage:doc:read` and `storage:doc:write` capabilities.
- `indexes` — List of field paths the adapter should index. See the Index Hints section below.
- `strict` — If `true`, reject documents with fields not in the schema. Default `false`.
- `extensions` — List of extension field paths this plugin adds to the collection (relevant when multiple plugins share the same CDM collection).
- `extension_schema` — Optional JSON Schema for the extension fields.
- `extension_indexes` — Extension fields the adapter should index.

## Validation Rules

`StorageContext` validates documents on **write operations only** (create, update, partial_update, and their batch variants):

1. Check that required base fields are present (for collections with a declared schema)
2. Validate the document body against the collection's JSON Schema (if one exists — whether CDM or custom)
3. Validate extension fields against the plugin's extension schema (if declared)
4. If strict mode is enabled, reject any fields not defined in the schema
5. If no schema is declared, skip validation entirely — store as-is

### Permissive by Default

When a schema is declared but `strict` is not set (or is `false`):

- Fields defined in the schema are validated against their declared types and constraints
- Fields not in the schema are accepted and stored without validation
- Extra fields are not indexed unless explicitly listed in `indexes`

This avoids breakage when a plugin updates its schema to add new fields — older documents that lack the field remain valid.

### Strict Mode

When `strict = true`:

- All fields must be defined in the schema
- Unknown fields cause `StorageError::ValidationFailed`
- Useful for plugins that need tight control over their data shape

### No Read Validation

Documents are not validated on read. Data is trusted once stored. This avoids performance overhead on the read path and prevents issues where a schema change would make previously valid documents unreadable.

## System-Managed Base Fields

Every document in every collection has these system-managed fields:

- `id` — String. Unique identifier. Generated by `StorageContext` on create if not provided by the caller.
- `created_at` — DateTime (ISO 8601). Set by `StorageContext` on create. Immutable after creation.
- `updated_at` — DateTime (ISO 8601). Set by `StorageContext` on every write operation.

These fields are managed by `StorageContext`, not by the plugin. A plugin can provide an `id` on create (useful for idempotent sync operations), but `created_at` and `updated_at` are always set by the system. This applies regardless of whether the collection has a declared schema.

## Extension Field Namespacing

Plugins attach extra fields to shared collection documents under a namespaced prefix:

```
ext.{plugin_id}.{field_name}
```

For example, a Google contacts connector might add:

```json
{
  "id": "abc-123",
  "name": "Jane Doe",
  "email": "jane@example.com",
  "created_at": "2026-03-28T10:00:00Z",
  "updated_at": "2026-03-28T10:00:00Z",
  "ext": {
    "connector-google": {
      "google_id": "people/12345",
      "last_sync": "2026-03-28T09:55:00Z"
    }
  }
}
```

Extension field rules:

- Each plugin can only read and write its own extension namespace
- Other plugins can see extension fields on read but cannot modify them
- Extension fields are stored as part of the document JSON, not in separate storage
- Extension fields can be validated against a schema if `extension_schema` is declared
- Extension fields can be indexed if listed in `extension_indexes`

## Index Hints

Index hints tell the document adapter which fields should be indexed for query performance. They are explicit in the manifest — `StorageContext` does not infer indexes from query patterns.

```toml
[collections.private.shopping_lists]
indexes = ["due_date", "category", "status"]
```

How adapters handle index hints:

- Adapters that support indexing (reported via capabilities) create indexes for the listed fields. For JSON-stored documents, this typically means generated columns or expression indexes.
- Adapters that do not support indexing ignore the hints and fall back to scanning.
- CDM recommended schemas ship with sensible default index hints (e.g., `created_at`, `updated_at`, and collection-specific fields like `start` for events, `due_date` for tasks).

Index hints are passed to the adapter via the `CollectionDescriptor` during `migrate`. New indexes added in a manifest update are created on the next startup.

## Schema Evolution

Schemas follow the additive-only rule within a major SDK version:

- **Non-breaking changes** — Add optional field, add enum value, add collection, relax a constraint. Applied automatically.
- **Breaking changes** — Remove or rename a field, change a field type, add a required field, remove an enum value, tighten a constraint. Require a major SDK version bump.

Plugin private collections follow the same rules. Migration transforms for breaking changes are declared in the manifest as ordered transform functions — see [[schema-versioning-rules]] for full details.

`StorageContext` executes transforms before calling `migrate` with the new descriptor, so by the time the adapter sees the new schema, data already conforms.
