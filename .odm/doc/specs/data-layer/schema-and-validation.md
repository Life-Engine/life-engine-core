# Schema and Validation Specification

## Schema Format

All schemas use JSON Schema draft 2020-12. Plugins ship schema files alongside their `manifest.toml`.

## Schema Declaration

Schemas are declared in two places:

- **CDM recommended schemas** — Shipped with the SDK and referenced via the `cdm:` prefix (e.g., `cdm:contacts`). See [[cdm-specification]].
- **Plugin collection schemas** — Declared in the plugin's `manifest.toml` under `[collections.<name>]`.

A collection declaration in `manifest.toml`:

```toml
[collections.contacts]
schema = "cdm:contacts"
access = "read-write"
extensions = ["ext.connector-google.google_id"]
extension_schema = "schemas/contact-extensions.json"
extension_indexes = ["ext.connector-google.google_id"]

[collections.shopping_lists]
schema = "schemas/shopping-list.json"
indexes = ["due_date", "category"]
strict = false

[collections.cache]
# schemaless — no schema field
```

Declaration fields:

- **`schema`** — Path to a JSON Schema file or a `cdm:` prefixed reference. Optional. Omitting makes the collection schemaless.
- **`access`** — Required. One of `read`, `write`, or `read-write`. Determines which operations the plugin can perform.
- **`indexes`** — A list of field paths to index. Optional. Passed as hints to the adapter.
- **`strict`** — Boolean, defaults to `false`. When `true`, unknown fields are rejected on write.
- **`extensions`** — A list of extension field paths this plugin declares. Optional.
- **`extension_schema`** — Path to a JSON Schema file for validating extension fields. Optional.
- **`extension_indexes`** — A list of extension field paths to index. Optional.

## Validation Rules

Schema validation applies to write operations only: `create`, `update`, `partial_update`, and their batch variants. Read operations are never validated.

Validation sequence on write:

1. Check that system-managed base fields (`id`, `created_at`, `updated_at`) are not set by the caller (except `id` on create, which is optional)
2. Validate the document body against the collection's JSON Schema, if one is declared
3. Validate extension fields against `extension_schema`, if declared
4. If `strict` mode is enabled, reject any fields not defined in the schema
5. If no schema is declared, skip validation entirely

Default behaviour is permissive — extra fields not in the schema are accepted and stored. Strict mode (`strict = true`) changes this to reject unknown fields with `StorageError::ValidationFailed`.

## System-Managed Base Fields

Every document in every collection carries these fields:

- **`id`** — A unique identifier. Generated by `StorageContext` on create if the caller does not provide one. Immutable after creation.
- **`created_at`** — Timestamp set by `StorageContext` on create. Immutable after creation.
- **`updated_at`** — Timestamp set by `StorageContext` on every write operation.

Callers must not set `created_at` or `updated_at`. `StorageContext` silently overwrites any caller-provided values for these fields.

## Extension Field Namespacing

Extension fields use the format `ext.{plugin_id}.{field_name}` and are stored inline in the document JSON.

Rules:

- Each plugin can only write to its own `ext.{plugin_id}` namespace
- Other plugins can read extension fields but must not modify them
- `StorageContext` enforces namespace isolation — writes to another plugin's namespace result in `StorageError::CapabilityDenied`
- Extension fields are included when reading documents (no filtering by default)

## Index Hints

Index declarations in `manifest.toml` are hints, not guarantees:

- Adapters that support indexing (reported via `AdapterCapabilities.indexing`) create the requested indexes during `migrate`
- Adapters without indexing capability silently ignore index hints
- CDM schemas ship default index hints for common query patterns
- Index hints are passed to the adapter via `CollectionDescriptor` during the `migrate` call

See [[document-storage-adapter]] for `CollectionDescriptor` and `FieldDescriptor` definitions.

## Schema Evolution

Schema changes must be additive-only within a major SDK version:

- Adding new optional fields is permitted
- Removing fields or changing field types is a breaking change
- Breaking changes require a new major SDK version

See [[schema-versioning-rules]] for full evolution rules.
